# Agent Skills 架构三部曲：从原理到企业级落地

> 本文基于真实技术面试对话整理，覆盖 Skill 本质、上下文控制、企业级安全三大核心维度。

---

## 引言

最近一场 AI Agent 平台的技术面试，面试官连续追问了 Skills 架构设计的核心问题。这些问题从原理到落地、从单 Skill 到多 Skill 并发、再到企业级安全，层次递进，恰好构成了 Agent Skills 架构设计的完整知识链。

整理出来，供做 AI Coding 工具或 Agent 平台的同学参考。

## 第一部：Skill 的本质与三层渐进式披露

### 1.1 Skill 解决什么问题？

**核心矛盾**：把所有知识塞进 System Prompt → 上下文过载 → 模型效果下降（"模型糊了"）

**Skill 的本质**：一个**结构化的知识包**，通过「渐进式披露」解决上下文稀缺性问题。

> 类比：你不会把整本字典塞进大脑，而是遇到生词时去查。Skill 就是 Agent 的「按需查字典」机制。

### 1.2 三层渐进式披露架构

```
┌─────────────────────────────────────────────┐
│              Skill 三层架构                  │
├─────────────────────────────────────────────┤
│  第一层：元数据层 (Metadata)                │
│  · name + description                      │
│  · 始终加载，让模型「知道有哪些能力」        │
│  · token 消耗：极少 (~50-100 token)        │
├─────────────────────────────────────────────┤
│  第二层：指令层 (SKILL.md 正文)            │
│  · 任务匹配时才加载                         │
│  · 包含完整的执行指令、约束、示例            │
│  · token 消耗：中等 (~3000-5000 token)     │
├─────────────────────────────────────────────┤
│  第三层：资源层 (scripts/references)        │
│  · 只有被主动引用时才进入上下文             │
│  · 包含可执行脚本、参考文档、数据集          │
│  · token 消耗：按需，可能很大               │
└─────────────────────────────────────────────┘
```

**设计原则**：模型只为「正在做的事」付 token。

### 1.3 为什么是三层？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 全量加载（单层） | 实现简单 | 上下文爆炸，token 成本高，模型容易混乱 |
| 两层（元数据+正文） | 减少无效加载 | 资源文件大的时候依然会撑爆上下文 |
| **三层（渐进式披露）** | **按需加载，token 效率最高** | 实现复杂度稍高，需要调度逻辑 |

## 第二部：多 Skill 并发的上下文控制

### 2.1 问题定义

用户任务同时触发 **3 个 Skill** → 上下文会爆炸吗？

```
场景：用户说"帮我分析这段代码，写成技术博客，发到公众号"

触发 Skill：
  · code-analyzer  (分析代码)
  · blog-writer     (写博客)
  · wechat-publisher (发公众号)

如果每个 Skill 的 SKILL.md 是 5000 token，
3 个并发 = 15000 token，这还没算上用户对话和历史上下文。
```

### 2.2 三层防御策略

#### 策略一：单 Skill token 上限约束

```
每个 SKILL.md 正文 ≤ 5000 token
→ 3 个并发 ≈ 15000 token
→ 在绝大多数模型的上下文窗口内仍可承受
```

**实践建议**：用 `tiktoken` 库在加载前做 token 计数，超标的 Skill 拒绝加载或触发裁剪。

#### 策略二：基于 description 的优先级权重

元数据的 `description` 字段本身就是**天然的优先级信号**：

```python
# 伪代码：Skill 匹配与排序
def rank_skills(task_description, available_skills):
    scores = []
    for skill in available_skills:
        # 用轻量模型或向量相似度计算相关性
        score = relevance_score(task_description, skill.description)
        scores.append((skill, score))
    return sorted(scores, key=lambda x: x[1], reverse=True)
```

模型会根据任务相关性**自动排序**，高相关 Skill 的完整指令层先加载，低相关的只保留元数据层。

#### 策略三：Skill 裁剪（Skill Pruning）

当确认超出 token 预算时，触发裁剪：

```
原始状态：
  Skill A: [元数据 + 指令层 + 资源层]
  Skill B: [元数据 + 指令层 + 资源层]
  Skill C: [元数据 + 指令层 + 资源层]

裁剪后（超出预算时）：
  Skill A (高优先级): [元数据 + 指令层(完整) + 资源层(按需)]
  Skill B (中优先级): [元数据 + 指令层(压缩为摘要) + 资源层(延迟)]
  Skill C (低优先级): [元数据(仅保留) + 指令层(移除) + 资源层(移除)]
```

**压缩指令层的方法**：
- 用小型摘要模型将 5000 token 的 SKILL.md 压缩为 200 token 的摘要
- 保留关键约束（do/don't）和输入输出格式
- 移除示例和详细解释

## 第三部：Skill 裁剪的回退机制

### 3.1 裁剪的风险

**问题**：裁剪掉关键 Skill 的指令层 → 模型执行出错 → 怎么补救？

### 3.2 执行验证环（Execution Verification Loop）

```
┌──────────────────────────────────────────────────────┐
│                 执行验证环                            │
│                                                      │
│  1. 模型输出执行计划 (Execution Plan)                │
│         ↓                                            │
│  2. 验证层检查 (人工 或 小模型快速校验)              │
│         ↓                                            │
│      ┌──────────────┐    ┌──────────────────────┐   │
│      │  校验通过      │    │  校验不通过           │   │
│      ↓              ↓    ↓                      │   │
│   真正执行        重跑 重新加载完整 Skill 上下文    │   │
│                  ↓                            │   │
│               执行计划 (修正后)                   │   │
│                  ↓                            │   │
│              再次校验                           │   │
│              └──────────────────────────────┘   │
└──────────────────────────────────────────────────────┘
```

### 3.3 验证层的两种实现

| 方式 | 适用场景 | 优点 | 缺点 |
|------|---------|------|------|
| **人工确认** | 高风险操作（删数据、发消息） | 安全性最高 | 破坏自动化体验 |
| **小模型校验** | 常规操作 | 自动化、快速（~100ms） | 需要高质量的校验 prompt |

**小模型校验 prompt 示例**：

```
你是一个执行计划校验器。请判断以下执行计划是否合理：

[Skill 指令层（裁剪后的摘要）]
{skill_summary}

[用户任务]
{user_task}

[模型输出的执行计划]
{execution_plan}

请回答：PASS（计划合理，可以执行）或 FAIL（计划有问题，需要重新加载完整上下文）。
如果 FAIL，请说明原因。
```

### 3.4 实测数据

> 实际项目测试：约 **5% 的执行需要回退**。这 5% 恰好是最容易出问题的场景——裁剪过度导致关键信息丢失。

**结论**：回退机制是有代价的（多一次模型调用），但这 5% 的安全边际非常值得。

## 第四部：企业级 Skill 市场的安全与质量管控

当你把 Skill 开放给第三方开发者，问题从「怎么设计架构」变成「怎么防止架构被滥用」。

### 4.1 安全三层防御

```
┌─────────────────────────────────────────────┐
│          企业 Skill 市场安全架构              │
├─────────────────────────────────────────────┤
│  第一层：沙箱隔离                           │
│  · 每个 Skill 在独立沙箱中执行              │
│  · 禁止访问主进程文件系统、网络              │
│  · 参考模型：浏览器 iframe 隔离              │
├─────────────────────────────────────────────┤
│  第二层：权限声明                             │
│  · Skill 元数据显式声明所需权限              │
│  · 域名白名单、端口、文件路径                │
│  · 加载前给用户确认（类似 App 安装权限）     │
├─────────────────────────────────────────────┤
│  第三层：质量审核                           │
│  · 静态分析：扫描代码和配置文件              │
│  · 动态测试：运行自动化测试用例              │
│  · 参考模型：App Store 审核流程              │
└─────────────────────────────────────────────┘
```

    # 文件系统隔离
    # 使用 chroot 或容器技术，限制文件访问范围

### 4.4 质量审核流程

```
提交 Skill
    ↓
[静态分析] 扫描恶意代码、权限过度申请、依赖漏洞
    ↓ 通过
[动态测试] 在隔离环境中运行官方测试用例
    ↓ 通过
[人工复审] 针对高风险 Skill（访问敏感权限）做二次审核
    ↓ 通过
上架 Skill 市场
```

**关键指标**：静态分析覆盖率 100%，动态测试用例通过率 100%，才能上架。

## 总结：Skills 架构设计的核心原则

| 原则 | 说明 |
|------|------|
| **渐进式披露** | 不为不用的知识付 token，三层架构是最优解 |
| **token 预算约束** | 单 Skill 有上限，多 Skill 有裁剪策略 |
| **验证与回退** | 裁剪有风险，执行验证环 + 5% 回退率是可接受成本 |
| **零信任沙箱** | 第三方 Skill 默认不可信，沙箱 + 权限声明 + 审核三层防御 |
| **参考成熟模型** | iframe 隔离、App Store 审核、tiktoken 计数，都是经过验证的方案 |
