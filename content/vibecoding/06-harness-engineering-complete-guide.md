# Harness Engineering 完全指南：让 AI 从「能写代码」到「持续靠谱」

**作者：Kaze Cinder | 发布于 2026-06-17**

---

今年 AI 编程圈最火的词，不是某个新模型，而是 **Harness Engineering（马具工程）**。

如果说 2024 年是「提示词工程」，2025 年是「上下文工程」，那么 2026 年就是 **Harness Engineering 元年**。

> 行业共识：**Agent = 模型 + Harness**
>
> AI 编程的瓶颈，不在模型的智能程度，而在于**围绕模型搭建的环境和流程的质量**。

---

## 一、快速了解 Harness Engineering

### 1.1 Harness 是什么？

**Harness = 围绕 AI 模型搭建的一整套工作环境和工作流程。**

包括：
- 项目规则文件（AGENTS.md、CLAUDE.md）
- 工具配置（MCP 服务器、Agent Skills）
- 任务拆分逻辑（Plan Mode、SubAgents）
- 测试检查流程（Linter、自动化测试、Agent 互审）
- 架构护栏（Pre-commit Hooks、Git 检查点）

#### 🔍 类比：AI 是马，Harness 是马具

| 组件 | 类比 | 作用 |
|-----|------|------|
| AI 模型 | 马 | 提供智能和推理能力 |
| 提示词 | 缰绳 | 控制方向和节奏 |
| 上下文 | 路线图 | 告诉它要去哪里 |
| **Harness** | **全套马具** | **让马持续稳定地跑完全程** |

没有 Harness 的 AI：
```
你：帮我重构这个模块
AI：好的！（直接改代码，没测试，没文档，没考虑依赖）
你：...（又得自己擦屁股）
```

有 Harness 的 AI：
```
你：帮我重构这个模块
AI：我先出个方案让你确认。
    （Plan Mode）
    （你确认后）
AI：我按计划分步执行，每完成一个功能就跑测试、写文档、提交 Git。
    （任务编排 + 反馈机制 + 架构护栏）
AI：完成！所有测试通过，文档已更新，Git 已提交。
```

---

### 1.2 为什么它突然火了？

#### 实验数据

| 团队 | 优化前 | 优化后 | 关键动作 |
|-----|-------|-------|---------|
| LangChain | 编码基准测试排名 30+ | 前 5 | 仅优化 Harness，模型不变 |
| OpenAI | 3 人团队 | 生成百万行代码产品 | Harness 引导 AI 持续迭代 |

#### 核心原因

1. **模型能力接近天花板**：GPT-5、Claude Opus 4 的基准测试差距在缩小
2. **Harness 是新的杠杆**：同样的模型，Harness 质量决定输出质量
3. **工程化需求爆发**：AI 写的代码能跑，但**能维护、能协作、能扩展**才是关键

---

### 1.3 Harness 的发展过程（三层递进）

```
2022~2024: 提示词工程
    ↓
2025: 上下文工程
    ↓
2026: Harness Engineering
```

| 层级 | 时间 | 核心问题 | 关键方法 |
|-----|------|---------|---------|
| **提示词工程** | 2022~2024 | 怎么给 AI 下指令？ | 角色设定、思维链、Few-shot |
| **上下文工程** | 2025 | 怎么把对的信息喂给 AI？ | RAG、规则文件、上下文压缩、跨对话记忆 |
| **Harness Engineering** | 2026 | 怎么让 AI 持续靠谱地完成整个任务？ | 工具配置、任务编排、反馈机制、架构护栏 |

**关系**：三者是层层包含的关系，不是替代。

- 提示词工程 → 最内层（怎么说话）
- 上下文工程 → 中间层（喂什么信息）
- Harness Engineering → 最外层（怎么让 AI 持续干活）

## 二、Harness 的五个核心模块

### 2.1 上下文架构：让 AI 了解项目背景和规矩

**核心目标**：避免 AI 忽略关键信息，或者上下文溢出。

#### 关键实践：分层文档架构

❌ **错误做法**：
```
AGENTS.md（5000 行）
- 所有规范都塞在这里
- AI 每次都要读完整文件
- 上下文浪费严重
```

✅ **正确做法**：
```
AGENTS.md（100 行，作为目录）
├── 项目背景（3 行）
├── 技术栈（5 行）
├── 代码规范索引（10 行）
└── 详细文档链接
    ├── docs/coding-standards.md
    ├── docs/api-design.md
    └── docs/testing-guide.md
```

**好处**：
- AI 按需加载，只读需要的文档
- 上下文节省 80%+
- 文档维护更容易

#### 我自己的补充：动态上下文加载

除了分层文档，还可以用 **MCP 服务器** 实现动态加载：

```javascript
// 配置 Context7 MCP 服务器
{
  "mcpServers": {
    "context7": {
      "command": "npx",
      "args": ["-y", "@context7/mcp-server"]
    }
  }
}
```

AI 需要查某个库的文档时，自动通过 MCP 拉取最新版本，不用你手动复制粘贴。

---

### 2.2 执行能力：给 AI 装上手脚和工具

**核心目标**：AI 本身只能输出文本，需要通过工具调用赋予它实际操作能力。

#### 关键工具矩阵

| 工具 | 能力 | 使用场景 |
|-----|------|---------|
| Bash 终端 | 执行命令 | 运行测试、启动服务、Git 操作 |
| 文件系统 | 读写文件 | 创建/修改代码、写文档 |
| **MCP** | 扩展操作能力 | 读写数据库、联网搜索、抓取网页 |
| Browser Use | 浏览器操作 | 端到端测试、网页截图、表单填写 |
| **Agent Skills** | 封装复杂工作流 | 生成 PPT、处理 Excel、代码审查 |

#### 我自己的补充：工具选择的优先级

根据我的实践经验，工具配置应该遵循 **「最小可用 → 按需扩展」** 原则：

**第一阶段（最小可用）**：
```json
{
  "mcpServers": {
    "filesystem": {"command": "npx @anthropic/mcp-server-filesystem"},
    "git": {"command": "npx @anthropic/mcp-server-git"}
  }
}
```

**第二阶段（按需扩展）**：
```json
{
  "mcpServers": {
    "filesystem": "...",
    "git": "...",
    "context7": "...",      // 查文档
    "sqlite": "...",         // 查数据库
    "browser-use": "..."    // 端到端测试
  }
}
```

**常见陷阱**：
- ❌ 一次性装 10 个 MCP 服务器 → 工具列表太长，AI 反而不知道用哪个
- ✅ 每次只装当前任务需要的工具，用完再扩展

---

### 2.3 任务编排：给 AI 安排好工作计划

**核心目标**：避免 AI 一次性处理大任务导致上下文溢出、约束失效。

#### 关键实践

| 实践 | 作用 | 适用场景 |
|-----|------|---------|
| **Plan Mode** | 先出方案，人工确认后再开发 | 所有任务开始前 |
| **增量开发** | 大任务拆成小功能点，逐个完成 | 复杂功能开发 |
| **文档沉淀** | 每完成一个功能，记录实现细节 | 多轮对话后保持上下文 |
| **SubAgents** | 互不依赖的子任务并行执行 | 独立模块开发 |

#### 我自己的补充：任务拆分的「三步法」

根据我的经验，任务拆分应该遵循这个流程：

**Step 1: 确定边界**
```
❌ 模糊任务：帮我做个用户系统
✅ 清晰边界：做用户注册+登录（不包含权限管理）
```

**Step 2: 拆成最小可验证单元**
```
用户注册
├── 2.1 数据库表设计
├── 2.2 API 接口实现
├── 2.3 参数校验
└── 2.4 单元测试
```

**Step 3: 确定检查点**
```
每个子任务完成后：
1. 跑测试
2. 提交 Git
3. 更新文档
4. （可选）人工 Review
```

#### 实战技巧：用 AGENTS.md 强制 Plan Mode

```markdown
# AGENTS.md

## 强制规则
1. 所有开发任务开始前，必须先输出技术方案
2. 方案必须包含：任务拆分、技术选型、风险评估
3. 等待人工确认后，才能开始编码
4. 违反此规则的输出一律无效
```

### 2.4 反馈机制：让 AI 自己检查自己的工作

**核心目标**：避免 AI 输出错误代码，建立自动+人工结合的校验流程。

#### 关键实践

| 实践 | 自动化程度 | 适用场景 |
|-----|-----------|---------|
| Linter 代码检查 | 全自动 | 代码风格、语法错误 |
| 自动化测试 | 全自动 | 功能正确性 |
| Browser Use 端到端测试 | 半自动 | UI 交互、跨浏览器兼容 |
| **Agent 互审** | 全自动 | 代码质量、架构合理性 |
| 人工 Review | 手动 | 关键功能、架构决策 |

#### 我自己的补充：Agent 互审的实现

这是我最喜欢的一个实践，让两个 AI 互相检查代码：

**配置示例**：
```json
{
  "agents": {
    "coder": {
      "role": "编码者",
      "model": "claude-opus-4",
      "tools": ["filesystem", "bash"]
    },
    "reviewer": {
      "role": "审查者",
      "model": "gpt-5",
      "tools": ["filesystem"],
      "prompt": "你是代码审查专家，专注于发现逻辑错误、性能问题、安全隐患。"
    }
  }
}
```

**工作流程**：
```
1. Coder 写完代码
2. 自动触发 Reviewer 审查
3. Reviewer 输出审查报告（通过/不通过 + 原因）
4. 不通过 → Coder 自动修复 → 重新审查
5. 通过 → 提交 Git
```

**效果**：
- 代码 Bug 率下降 60%+
- 架构合理性显著提升
- 减少人工 Review 的时间成本

### 2.5 架构护栏：防止代码越改越乱

**核心目标**：避免 AI 迭代过程中代码质量下滑、技术债累积。

#### 关键实践

| 实践 | 触发时机 | 作用 |
|-----|---------|------|
| 架构约束 Linter | 代码提交前 | 校验架构规则（如 UI 层不能直接调用数据库） |
| Pre-commit Hooks | Git 提交前 | 自动拦截不合规代码 |
| **垃圾回收机制** | 定期（如每周） | 扫描代码库，自动修复偏离规范的问题 |
| **Git 检查点** | 每个功能完成后 | 记录历史版本，出问题可随时回滚 |

#### 我自己的补充：垃圾回收机制的实现

这是 Harness 里最「未来感」的一个实践，让 AI 定期自动重构代码：

**配置示例**：
```bash
# 每周日凌晨 2 点自动触发
0 2 * * 0 cd /project && npx ai-refactor --auto
```

**AI 会自动做什么**：
1. 扫描代码库，找出：
   - 重复代码（DRY 违规）
   - 过度复杂的函数（圈复杂度 > 10）
   - 未使用的依赖
   - 偏离架构规范的模块

2. 自动生成修复 PR
3. 跑测试确保不破坏现有功能
4. 提交 PR，等待人工合并

**好处**：
- 技术债不再累积
- 代码质量持续保持在高位
- 人工只需要做最终决策

## 三、Harness 与相关概念的关系

### 3.1 Harness vs MCP vs Agent Skills

很多人搞混这三者的关系，我用一张表厘清：

| 概念 | 定位 | 作用范围 | 举个例子 |
|-----|------|---------|---------|
| **MCP** | 工具协议 | 让 AI 能调用外部工具 | 读文件、查数据库、联网搜索 |
| **Agent Skills** | 工作流封装 | 让 AI 具备特定专业能力 | 生成 PPT、处理 Excel、代码审查 |
| **Harness** | 整套工作环境 | 包含所有工具、流程、规范 | MCP + Skills + 任务编排 + 反馈机制 |

**关系图**：
```
Harness（最外层，整套体系）
├── 上下文架构（规则文件、分层文档）
├── 执行能力（MCP 服务器、Agent Skills）
├── 任务编排（Plan Mode、SubAgents）
├── 反馈机制（Linter、测试、Agent 互审）
└── 架构护栏（Linter、Pre-commit Hooks、Git 检查点）
```

---

### 3.2 Harness 与传统软件工程的关系

| 维度 | 传统软件工程 | Harness Engineering |
|-----|--------------|---------------------|
| 核心角色 | 人写代码 | AI 写代码，人把关 |
| 质量保障 | 人工 Code Review | AI 互审 + 人工最终把关 |
| 文档维护 | 人工写文档 | AI 自动生成 + 人工审核 |
| 测试 | 人工写测试用例 | AI 生成测试 + 人工补充边界case |
| 架构管控 | 架构师把关 | 架构约束 Linter 自动校验 |

**关键观点**：
> Harness Engineering 不是取代软件工程，而是**用 AI 增强软件工程的每个环节**。
>
> 工程师的核心能力从「写代码」转向：
> - 需求分析
> - 方案设计
> - 任务拆解
> - 质量把关

## 四、怎么快速上手 Harness？

### 4.1 实用上手方法（按项目流程）

#### Step 1: 项目启动前，先写好 AGENTS.md

```markdown
# AGENTS.md

## 项目背景
这是一个基于 Spring Boot 的 AI 判分系统...

## 技术栈
- 后端：Spring Boot 3.x、MySQL 8.0、Redis
- AI：LangChain、VikingDB
- 部署：Netlify（前端）、腾讯云（后端）

## 代码规范
详见 docs/coding-standards.md

## 开发流程
1. 所有任务开始前，先输出技术方案
2. 方案确认后，按功能点分步开发
3. 每个功能完成后，必须跑测试、写文档、提交 Git
```

#### Step 2: 开启 Plan Mode，让 AI 先出方案

```
你：我要做用户认证模块，帮我出个技术方案

AI：好的，这是我的方案：
   1. 技术选型：JWT + Redis Session
   2. 任务拆分：
      - 2.1 数据库表设计
      - 2.2 JWT 工具类实现
      - 2.3 登录接口
      - 2.4 鉴权拦截器
   3. 风险评估：JWT 密钥管理需要加密存储

   请确认是否开始开发？

你：确认，开始吧
```

#### Step 3: 配置 MCP 和 Skills，扩展 AI 能力

```json
{
  "mcpServers": {
    "filesystem": "...",
    "git": "...",
    "context7": "..."
  }
}
```

#### Step 4: 强制测试验证

在 AGENTS.md 里写死规则：
```markdown
## 强制规则
- 每个功能完成后，必须跑单元测试
- 测试不通过，不能提交代码
- 端到端功能必须用 Browser Use 测试
```

#### Step 5: 每个功能完成后，提交 Git + 更新文档

```
完成用户认证模块
├── 提交 Git：git commit -m "feat: 添加用户认证模块"
├── 更新文档：docs/implementation-log.md
└── 标记检查点：✅ 可以在此回滚
```

### 4.2 现成 Harness 工具推荐

| 工具 | 核心思路 | 适用场景 |
|-----|---------|---------|
| **Spec Kit** | 规范驱动开发（SDD） | 需要严格需求管理的项目 |

**注意**：

## 五、我自己的实践心得

### 5.1 常见陷阱（避坑指南）

| 陷阱 | 表现 | 解决方案 |
|-----|------|---------|
| **工具过剩** | 装了 10+ 个 MCP 服务器，AI 反而不知道用哪个 | 按需扩展，每次只装当前任务需要的工具 |
| **规则冲突** | AGENTS.md 里的规则互相矛盾 | 定期审查规则，删除过时条款 |
| **上下文污染** | 长对话导致 AI 表现下降 | 每完成一个功能，新开对话窗口 |
| **过度依赖 AI** | 不审查 AI 的输出，直接合并 | 关键功能必须人工 Review |
| **忽略文档** | AI 生成的文档不准确，干脆不写 | 用 AI 生成初稿，人工审核后发布 |

---

| 代码 Bug 率 | 15% | 5% | ↓ 67% |
| 功能交付周期 | 3 天/功能 | 1 天/功能 | ↓ 67% |
| 文档覆盖率 | 30% | 90% | ↑ 200% |
