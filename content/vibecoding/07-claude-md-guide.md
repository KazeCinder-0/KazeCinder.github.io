# CLAUDE.md 完全指南：给 AI 写一份会自动加载的入职手册

来源：程序员鱼皮（公众号）| 整理：Kaze Cinder

---

## 问题的根源

用了一段时间 Claude Code、Cursor 之后，很多人开始抱怨：AI 怎么总是不听话？让它改个 Bug 反而引入三个新 Bug，让它加个功能结果技术栈都给换了。

**问题的根源不是 AI 笨，而是你没有给它足够的上下文。**

AI 每次开始新会话，都是一张白纸。它不知道你的项目用的是什么技术栈，不知道你们团队的代码规范，不知道你之前做了什么决策。你不说，它就猜。猜对了算你运气好，猜错了就得返工。

那有没有一种方法，让 AI 每次开始工作前就自动了解你的项目？有，这就是 **CLAUDE.md**。

---

## 一、CLAUDE.md 是什么

CLAUDE.md 是放在项目根目录下的一个 Markdown 文件。Claude Code 每次启动会话时，都会**自动读取**这个文件，把里面的内容当作项目的背景知识。

可以把它理解为**给 AI 写的「入职手册」**。就像新员工入职第一天，你会给他一份文档，介绍公司用什么技术、代码怎么写、测试怎么跑、有什么坑要注意。CLAUDE.md 干的就是这个事。

一个最简单的 CLAUDE.md 长这样：

```markdown
# 我的博客系统

Next.js 16 + TypeScript + Tailwind CSS + Supabase

## 常用命令
- npm run dev        # 启动开发服务器
- npm run test       # 跑测试
- npm run build      # 生产构建

## 代码规范
- 使用函数式组件，不用 class 组件
- 样式只用 Tailwind CSS，不写自定义 CSS
- 所有组件必须有 TypeScript 类型定义
```

AI 每次开始工作前都会先读一遍，然后所有生成的代码都会自动遵循这些规范，不用你每次都重复交代。

> **注意**：CLAUDE.md 对 AI 来说**是建议**而**不是强制执行**的配置。它的效果取决于你写得有多具体、多简洁。越精确的指令 AI 遵循得越好，写了一大堆模糊的要求反而可能被选择性忽略。

---

## 二、更通用的 AGENTS.md

你可能会想，这是 Claude Code 的专属功能吗？用 Cursor 怎么办？

几乎所有主流 AI 编程工具都有类似的机制。必须知道的是 **AGENTS.md**，它是一个**跨工具的开放标准**，由 OpenAI、Google、Cursor 等多家公司联合推出，目前已有上万个开源项目在用，支持 30 多种 AI 编程工具。

Claude Code 虽然默认只读 CLAUDE.md，但你可以在 CLAUDE.md 里用一行 `@AGENTS.md` 把它导入，两边就打通了。

AGENTS.md 的用途也不只限于项目开发指导——你可以用它定义各种工作流，像图文创作、产品客服、数据分析这些场景，都可以用一份 Markdown 文件把 AI 的工作方式固定下来，每次启动自动加载。

---

## 三、为什么一定要写好这个文件

你可能觉得，不就是个配置文件吗，随便写写不就行了？大错特错。

CLAUDE.md 的质量**直接决定了 AI 帮你干活的效率和准确度**。一个好的 CLAUDE.md 至少能解决三个问题：

| 问题 | 说明 |
|------|------|
| **减少重复沟通** | 每次新会话无需重复交代技术栈和规范，AI 自动就知道了 |
| **提升代码一致性** | 不再混用函数组件/class 组件、CSS Modules/内联样式 |
| **降低出错率** | 项目特有的约定（如 API 响应格式）固化后 AI 自动遵循 |

有个关键数据值得记住：

> 规则文件在 **50 行**左右时，AI 遵循率可以达到 **94%**；但如果一股脑堆到 **400 行**，遵循率就掉到了 **71%**。

**CLAUDE.md 不是越多越好，而是要精准。**

---

## 四、怎么写好 CLAUDE.md（8 个核心技巧）

### 1. 控制在 200 行以内

这是 Claude Code 官方文档给出的建议。

CLAUDE.md 的内容会在每次会话开始时加载到 AI 的上下文窗口。这个窗口是有限的，规则文件越长，留给实际工作内容的空间就越少。AI 对指令的遵循能力与指令数量负相关——大概能合理遵循 150 到 200 条指令，超过后遵循质量会**全面均匀下降**，而不只是忽略新加的。

写 CLAUDE.md 要像写简历一样，精炼再精炼。对每一行，问自己：**删掉它 AI 会犯错吗？** 不会，就果断删掉。

---

### 2. 只写 AI 猜不到的东西

官方文档给了一个清单：

**✅ 应该写进去的：**
- AI 不可能猜到的构建命令和测试命令
- 跟默认不同的代码风格规则
- 团队的 Git 规范（分支命名、PR 格式）
- 项目特有的架构决策
- 开发环境的特殊要求（必需的环境变量等）
- 容易踩坑的地方

**❌ 不该写进去的：**
- AI 读代码就能知道的东西
- 通用的语言规范（AI 本来就会）
- 详细的 API 文档（放别的地方，这里放链接就行）
- "写干净的代码"、"注意性能"这种废话

---

### 3. 用正面指令代替否定指令

这个技巧有点反直觉。研究发现，87.5% 的 AI 规则违反都来自同一个原因：**否定句反而把被禁止的概念激活了**。你写「不要使用分号」，AI 看到"分号"这个词，反而更容易在后续生成中产出分号。

这跟心理学里的**白熊效应**一模一样——叫你不要想白熊，脑子里马上全是白熊。

| ❌ 否定写法 | ✅ 正面写法 |
|-----------|-----------|
| 不要使用 class 组件 | 使用函数式组件和 Hooks |
| 不要用 any 类型 | 所有变量必须有明确的 TypeScript 类型定义 |
| 不要在主分支直接提交 | 所有变更通过 feature 分支提交 PR |

---

### 4. 指令要具体可验证

模糊的指令等于没有指令。AI 很擅长处理具体的规则，但对抽象的要求会自行发挥。

| ❌ 模糊 | ✅ 具体 |
|--------|--------|
| 正确格式化代码 | 使用 2 空格缩进 |
| 测试你的更改 | 提交前运行 `npm test` |
| 保持文件有序 | API handlers 放在 `src/api/handlers/` |

越具体的指令，AI 越容易判断自己有没有遵循，违反了也更容易被发现。

---

### 5. 用 Hooks 强制执行关键操作

CLAUDE.md 本质上只是建议，不是 100% 生效的禁令。

像代码格式化、提交前跑测试、危险命令拦截这些任务，应该用**确定性的自动化脚本**来执行。Claude Code 提供了 Hooks 机制，可以在 AI 操作的关键节点自动执行你预设的脚本，AI 根本不介入，100% 会执行。

**一句话总结：CLAUDE.md 负责建议，Hooks 负责强制。**

---

### 6. 大项目用路径规则按需加载

如果项目比较大，200 行根本装不下所有规范。这时不要硬往 CLAUDE.md 里塞，而是用 `.claude/rules/` 目录来拆分。

每个规则文件可以用 YAML frontmatter 指定只对特定路径生效：

```yaml
---
paths:
  - "src/api/**/*.ts"
---

# API 开发规范
- 所有 API 端点必须包含输入验证
- 使用标准的错误响应格式
- 添加 OpenAPI 文档注释
```

这样只有当 Claude 操作对应目录下的文件时，这些规则才会加载到上下文中，既省空间，又不用无关规则干扰 AI。

---

### 7. 用 @import 做渐进式披露

社区里最受推崇的高级技巧叫**渐进式披露（Progressive Disclosure）**，核心思想：不要把所有东西塞进 CLAUDE.md，而是**告诉 AI 去哪里找它需要的信息**。

做法很简单。在项目中创建一个 `docs/` 目录，放各类专题文档，然后在 CLAUDE.md 中用简短描述加触发条件来引用：

```markdown
## 参考文档

### API 架构 — @docs/api-architecture.md
何时阅读：添加或修改 API 端点时

### 数据库设计 — @docs/database-design.md
何时阅读：创建或修改数据模型时
```

这样 Claude 只在真正需要的时候才去读取对应文档，平时不占上下文空间。`@path` 导入语法支持相对路径和绝对路径，最多可以嵌套 4 层。

Anthropic 官方把这个策略叫 **just-in-time**——让 AI 按需获取信息，而不是全量加载。这个思路和之前聊过的 [Agent Skills 架构](./04-agent-skills-architecture.md)本质上是一样的。

---

### 8. 让 AI 帮你维护

CLAUDE.md 不是写好了就不管了，它应该是一份**随着项目成长不断完善的"活"文档**。

Claude Code 的创始人推荐一个习惯：**每次 Claude 犯错，不要只修正错误，还要顺手让它把纠正写进 CLAUDE.md。**

比如 Claude 用了错误的导入路径，纠正完之后跟它说一句：「把这条规则更新到 CLAUDE.md 里，下次别再犯。」Claude 特别擅长给自己写规则，时间久了，你的 CLAUDE.md 就变成了一份完整的项目知识库，出错率会明显下降。

另外，Claude Code 现在还有 **Auto Memory** 自动记忆功能，Claude 会自动记录它在工作中发现的模式和偏好，存到 `~/.claude/projects/<项目>/memory/` 目录下。你随时可以用 `/memory` 命令查看和编辑这些记忆。

## 五、如何快速创建

### 方法一：用 `/init` 命令自动生成

在 Claude Code 中输入 `/init`，Claude 会自动分析你的代码库，生成一份包含构建命令、测试指令和项目规范的 CLAUDE.md。

> ⚠️ 注意：自动生成的内容需要自己过一遍，该删的删、该补的补。一条不好的指令产生的破坏力远超一行坏代码。

### 方法二：从对话中沉淀

先不急着写 CLAUDE.md，正常跟 AI 协作完成一两个任务。做完之后，让 AI 回顾整个对话过程，帮你总结出需要记住的项目规范和注意事项，直接写成 CLAUDE.md。这样得到的规则文件是从实战中提炼出来的，比凭空想象精准得多。

### 方法三：参考优秀的开源项目

GitHub 上有个 `awesome-claude-md` 仓库，收录了 100 多个真实项目的 CLAUDE.md 示例，覆盖各种技术栈，包括 Anthropic 官方的、Cloudflare 的 monorepo 项目等等。还有个 `awesome-claude-code` 仓库，整理了 Skills、Hooks、斜杠命令等 Claude Code 生态工具。

**记住：CLAUDE.md 的内容宁缺毋滥。先从最关键的 20 行开始，在实际使用中再慢慢补充。**

## 六、多层级配置与团队协作

### 四层作用域

CLAUDE.md 不只能放在项目根目录，它支持四个层级：

| 层级 | 路径 | 说明 |
|------|------|------|
| 组织级 | 由 IT 统一部署 | **不可被个人或项目覆盖**，公司底线规范 |
| 用户级 | `~/.claude/CLAUDE.md` | 对本机所有项目生效 |
| 项目级 | `./CLAUDE.md` | 只对当前项目生效，**可提交到 Git** |
| 本地级 | `./CLAUDE.local.md` | 只对当前项目生效，**不提交到 Git** |

优先级：**本地级 > 项目级 > 用户级**。建议团队共享的规范写在项目级，个人偏好写在本地级，互不干扰。

### 团队共建建议

用 Git 管理 CLAUDE.md，让团队成员共同维护。每当 Code Review 中发现 AI 犯了什么错误，就把修正追加进去。时间久了，这个文件就沉淀了团队的开发经验和规范，变成整个团队的 AI 使用手册，新人入职也能快速上手。

### 重要规则一定要存文件

当你用 `/compact` 压缩上下文时，CLAUDE.md 会被重新读取和注入，但你在对话中口头说的指令压缩后就没了。**凡是重要的规则，一定要写进文件，不要只靠 AI 的短期记忆。**

---

## 七、排查规则不生效的问题

如果发现 AI 没在遵守 CLAUDE.md 里的规则，按这个顺序排查：

1. 运行 `/memory` 命令，看看 CLAUDE.md 有没有被正确加载
2. 检查指令是否太模糊（"格式化好代码" → "使用 2 空格缩进、单行不超过 80 字符"）
3. 检查不同文件之间有没有矛盾的指令
4. 如果某个操作必须强制在特定时机执行，改用 **Hooks** 而不是 CLAUDE.md

---

## 总结

| 维度 | 建议 |
|------|------|
| 长度 | 控制在 200 行以内 |
| 内容 | 只写 AI 猜不到的东西 |
| 写法 | 用正面指令，具体可验证 |
| 强制操作 | 用 Hooks，不要靠 CLAUDE.md |
| 大项目 | 用路径规则和 @import 按需加载 |
| 维护 | 犯一次错，更新一次规则 |
| 团队协作 | 用 Git 管理，Code Review 时共建 |

以前我们写 README.md 是给人看的，现在写 CLAUDE.md 是给 AI 看的。本质一样：把项目的重要信息组织好、传达清楚。区别在于，给 AI 写的说明书要更精炼、更具体、更可执行。

**从最简单的 20 行开始，写技术栈、构建命令、以及 AI 总是犯错的那几个规则。你会立刻感受到它的强大。**
