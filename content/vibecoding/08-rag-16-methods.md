# RAG 完全指南：16 种 RAG 方案一次讲清

来源：程序员鱼皮（编程导航）| 整理：Kaze Cinder

---

## 一、什么是 RAG

### AI 大模型的三个硬伤

1. **知识有截止日期**——训练数据存在时间断层
2. **一本正经地胡说八道**——幻觉问题
3. **缺乏私有知识**——无法访问企业内部文档

### RAG 的核心思想

**先搜再答**：让大模型在回答之前先去搜索相关资料，再基于搜到的知识来组织答案。就像考试时遇到不会的题先翻书，再根据书里的知识答题。

### 为什么有长上下文窗口还需要 RAG？

把所有文档塞进上下文窗口**既贵又不靠谱**：

- 上下文越长，token 费用越高
- **Lost in the Middle 问题**：大模型对超长上下文中间部分的注意力会明显下降
- 最佳实践：先用 RAG 提供精确资料，再利用长上下文窗口做分析推理，两者互补

## 二、16 种主流 RAG 方案

### （一）标准 RAG 及变体

#### 1. Naive RAG（朴素 RAG）

最基础的实现，核心概念是**向量**——把一段文字用一串数字表示，让计算机可以比较语义相似度。

**执行流程：**

- **离线索引**：文档切块 → Embedding 转向量 → 存入向量数据库（Milvus、Chroma、Qdrant 等）
- **在线问答**：用户问题转向量 → 向量库搜索 Top K → 拼成 Prompt → 交给 LLM 生成

**存在的问题**：切块粗暴可能截断语义、检索质量完全依赖 Embedding、无法识别噪声文档。

---

#### 2. Multi-Query RAG（多查询 RAG）

**解决问题**：用户提问方式千奇百怪，措辞与文档术语差距大时，向量检索可能搜不到正确内容。

**核心思路**：让 LLM 把原始问题改写成多种表述，分别搜索，合并去重结果。

**代价**：每次多调一次 LLM 做改写，延迟和成本增加。适用面向普通用户的客服、电商场景。专业术语规范领域收益有限。

---

#### 3. HyDE（假设性文档嵌入）

**解决问题**：用户问题很短，文档描述很长，在 Embedding 空间距离远。

**核心思路**：让 LLM 凭空写一个"假答案"（不必准确），用假答案的向量去检索。因为假答案和真实文档文体更接近，向量空间距离也更近。

**风险**：如果假答案方向跑偏，检索结果会更差。冷门领域慎用。

---

### （二）提升检索质量

#### 4. 语义分块（Semantic Chunking）

**解决问题**：固定字数切块会截断完整语义。

**核心思路**：按句子拆开文档，计算相邻句子 Embedding 相似度，相似度骤降说明话题变了，在此处切一刀。

**代价**：每句话都算一次 Embedding，成本更高。适用会议纪要、访谈等松散文档；有清晰章节结构的文档直接按标题切即可。

---

#### 5. 层级索引（Parent-Child Retrieval）

**解决问题**：小块检索精度高但上下文不足，大块上下文丰富但噪声也多。

**核心思路**：文档切成大块，每大块细分小块。检索时小块匹配，命中后返回所属大块。类比：搜到某句话，把整个章节都拿来看。

**适用场景**：技术手册、法律合同等长文档。

---

#### 6. Hybrid Search 混合检索

**解决问题**：纯向量无法精确匹配术语（如 `ERROR_CODE_4012`），纯关键词不理解语义（搜"如何退款"找不到"退货及返还货款流程"）。

**核心思路**：同时使用向量搜索（语义）+ BM25 关键词搜索（精确），通过 **RRF（倒数排序融合）** 算法合并排序。

**结论**：几乎所有生产环境都建议用 Hybrid Search 替代纯向量搜索，尤其术语密集领域。

---

#### 7. Reranking 精排

**解决问题**：检索召回的候选文档中总混着噪声。

**核心思路**：在检索和生成之间加精排步骤，用 Reranker 模型给每对 (query, doc) 重新打分。

**生产环境级联检索方案（十万级语料库）：**
- 第一层：粗检索召回 150 个（保召回率）
- 第二层：轻量 Reranker 初筛，保留 20 个
- 第三层：Cross-Encoder 精排，选出 Top 5

---

### （三）RAG 反思机制

#### 8. Corrective RAG（CRAG，纠正式 RAG）

**核心思路**：在检索和生成之间插一个"质检员"，逐个审查文档相关性：
- 打分高 → 直接喂给 LLM 生成
- 打分低 → 回退到 Web 搜索兜底
- 打分模糊 → 两边合并

---

#### 9. Self-RAG（自我反思 RAG）

**核心思路**：设置四个检查点，每一步让模型自我审视：
1. **Retrieve**：这个问题需要检索吗？
2. **IsRel**：检索到的文档相关吗？
3. **IsSup**：我的回答有文档支撑吗？（价值最高）
4. **IsUse**：这个答案对用户有用吗？

---

#### 10. Adaptive RAG（自适应 RAG）

**核心思路**：在前面加一个路由器（分类器），判断问题复杂度，按需分流：
- 简单问题（"今天星期几"）→ 直接回答
- 一般问题 → 检索一次
- 复杂问题 → 走完整 CRAG 流程

分类器可微调小模型或用 LLM few-shot 实现。适用流量混杂的场景。

---

### （四）结构化知识增强

#### 11. GraphRAG（图谱 RAG）

**解决问题**：答案散落在多个文档中，需要串联推理（跨文档多跳）。

举例：文档 A 说"张三是 AI 部门负责人"，文档 B 说"AI 部门属于技术中心"，需要推理出"张三属于技术中心"。

**提出方**：Microsoft Research，2024 年

**核心步骤**：LLM 抽取实体和关系构建知识图谱 → Leiden 算法社区划分并生成摘要 → 提问时沿关系获取子图。

**特点**：全局语义理解显著优于传统向量 RAG，但构建成本高，使用前务必评估是否必要。

---

#### 12. Text-to-SQL RAG（文本转 SQL RAG）

**解决问题**：结构化表格数据做 Embedding 非常低效，向量搜索对聚合、排序、筛选完全没招。

**核心思路**：LLM 把自然语言翻译成 SQL，执行查询，查询结果作为上下文回答。

**适用场景**：BI 看板问答、数据库运维助手、财务报表查询。**生产环境需配备只读权限、SQL 审计、沙盒隔离等安全措施。**

---

### （五）智能体驱动 RAG

#### 13. Agentic RAG（智能体 RAG）

**核心思路**：一个 AI Agent 自动调度，配备一组检索工具（向量、Web、SQL、图谱等），进入 **ReAct 循环：思考 → 行动 → 观察**。信息不够就换方式再搜，够了就生成回答。

**现状**：已是较主流的生产范式，Cursor 用的就是这种方式。

---

#### 14. Multi-Agent RAG（多智能体 RAG）

**核心思路**：拆分为多个专职 Agent：
- Router Agent：负责分发
- 各 RAG Agent：负责对应领域检索和推理
- Verification Agent：负责质检
- Generation Agent：负责润色输出

**适用场景**：数据源多、权限复杂、语言多样的企业级知识库。

---

### （六）RAG 能力扩展

#### 15. 多模态 RAG（Multimodal RAG）

**解决问题**：企业文档中大量图表、流程图、架构图等，纯文本 RAG 会丢失这些信息。

**核心思路**：图片、表格和文本统一到一个向量空间，生成阶段用**视觉语言模型（VLM）** 处理混合模态上下文。

---

#### 16. Speculative RAG（推测性 RAG）

**解决问题**：所有文档塞同一个 Prompt 增加推理延迟，噪声文档会带偏生成。

**核心思路**（借鉴 Speculative Decoding）：召回多个候选文档 → 分成子集 → 多个专家小模型并行生成草稿 → 强模型做一次验证选出最佳。

类比：多人并行干活，PM 最后简单验证即可。

---

## 三、方法选择指南

| 你的情况 | 推荐方案 |
|---------|---------|
| 标准文本知识库，追求基本可用 | Naive RAG |
| 用户提问风格多变、口语化 | Multi-Query RAG 或 HyDE |
| 生产环境，追求检索质量 | Hybrid Search + Reranking |
| 对准确率要求高，不能容忍幻觉 | Corrective RAG 或 Self-RAG |
| 查询复杂度差异大 | Adaptive RAG 路由 |
| 需要跨文档多跳推理 | GraphRAG |
| 数据以结构化表格为主 | Text-to-SQL RAG |
| 文档包含大量图表/图片 | 多模态 RAG |
| 多数据源、多类型混合 | Agentic RAG / Multi-Agent RAG |
| 延迟敏感 | Speculative RAG |

### 给初学者的建议

> **从简单开始，逐步完善。** 先跑通 Naive RAG，发现哪个环节出问题，再针对性地选用对应方案。千万别一上来就搞 Multi-Agent + GraphRAG + 多模态全家桶——实现成本高，效果也不一定更好。

## 四、RAG 评估框架——RAGAS

| 指标 | 含义 |
|------|------|
| 忠实度 | 回答有没有瞎编 |
| 答案相关性 | 答的是不是你问的 |
| 上下文精确率 | 搜到的有多少是有用的 |
| 上下文召回率 | 该搜到的搜到了吗 |

**先评估效果再优化**，不要凭感觉调参数。

## 总结

16 种方案覆盖六大方向：

| 方向 | 方案 |
|------|------|
| 标准检索与变体 | Naive / Multi-Query / HyDE |
| 检索质量提升 | 语义分块、层级索引、混合检索、精排 |
| 反思纠错 | CRAG、Self-RAG、Adaptive RAG |
| 结构化增强 | GraphRAG、Text-to-SQL RAG |
| 智能体驱动 | Agentic RAG、Multi-Agent RAG |
| 能力扩展 | 多模态 RAG、Speculative RAG |

**核心方法论：从简单开始，逐步完善，根据实际场景选择合适方案。**
